# Not Alone Studios — Product‑Led Hybrid Website Blueprint (v2, Expanded)

> **Intent:** A product‑first website that makes Not Alone Studios feel like a **credible product company** (ZenOps/Revalk front‑and‑center), while keeping a **selective, premium Studio lane** for revenue + credibility.  
> **North Star:** Visitors should understand in **5 seconds**: **what you build**, **for whom**, **why it’s different**, and **what to do next**.

---

## 0) Executive Summary (What to build)

### The site should feel like:
- A **Systems Product House** (not a generic agency)
- Calm, premium, **evidence‑first**, “operator‑grade”
- Products are the hero; Studio services are a **scarce secondary option**

### What we are selling:
- **ZenOps** (Primary): “Deterministic valuation operations + billing gates + audit trail”
- **Revalk** (Secondary): “Premium study OS that converts doubt → recall → score”
- **Studio lane** (Secondary): “We build operating systems for a few clients per quarter”

### The parent brand promise (one line)
**Not Alone Studios builds evidence‑first operating systems that turn chaos into repeatable output.**

### The unifying doctrine (anchor everywhere)
**Capture → Enforce → Release**  
(or product-specific: **Evidence → Rules → Readiness → Release**)

## 6.3 Feature pillars (A–F) (from your pack)
### Pillar A — Factory Workflow
- assignment lifecycle/state machine
- tasks tied to work orders
- SLA and overdue signals  
**Value:** fewer dropped balls, faster turnaround, clear accountability.

### Pillar B — Evidence-First Capture
- docs/photos/screenshots linked to assignments
- evidence checklists (by bank/report type)
- field-to-evidence linking
- annexure ordering metadata  
**Value:** fewer rejections, safer defensibility, smoother QC.

### Pillar C — Rules + Readiness Engine
- consistent derived values (95% realizable, 80% distress etc.)
- adopted vs market inversion logic
- unit standardization
- rounding rules
- readiness blocks release until complete  
**Value:** consistent reports, predictable outcomes.

### Pillar D — Repogen Factory Model
- commissioned work (tenant/external/channel)
- production queue execution
- pack/job/artifact versioning + export hash metadata
- manual “Release Deliverables” gate with billing enforcement  
**Value:** scalable report production service without losing control.

### Pillar E — India-friendly Billing
- postpaid invoicing + payment proof + mark-paid + audit
- credits: reserve → consume → release
- subscription refills managed in Studio
- reconciliation timelines  
**Value:** faster collections, fewer disputes.

### Pillar F — Audit Trail & Versioning
- immutable snapshots (contract + derived + readiness)
- job/artifact version history
- release records with idempotency keys + override reasons
- strict RBAC + row-level isolation  
**Value:** bank comfort + legal defensibility.

2) Audit-grade snapshots + field-to-evidence links  
3) Factory mode for commissioned work  
4) Billing gates built for service reality (credits/postpaid/payment proof)

---

## 6.6 Workflows (How it works)
### Workflow 1 — Standard job
1) Create work order  
2) Upload/link evidence  
3) Fill contract fields  
4) Rules compute derived values + warnings  
5) Readiness confirms completeness  
6) Create pack/job  
7) Manual release (billing gate enforced)  
8) Close with audit record  

### Workflow 2 — Commissioned job
1) Channel submits request  
2) Accept → reserve credits OR invoice  
3) Evidence intake + field linking  
4) Readiness → pack/job  
5) Release deliverables → consume credits OR require paid invoice
